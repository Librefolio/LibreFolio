# F-012 â€” BRIM Framework (Broker Report Import)

## What it does
A pipeline for importing broker transaction reports. Users upload a file, the system detects or selects the appropriate plugin, parses the file, reviews the extracted transactions in a staging area, resolves unknown assets, and commits to the database.

## Layer breakdown
**Backend**: `backend/app/services/brim_provider.py` (abstract base), `brim_providers/` (11 plugins)
**Frontend**: `frontend/src/lib/components/brokers/` (BrokerImportFilesModal, AssetMatchingWizard)

## Implementation notes

### Fake Asset ID Flow (key design)
BRIM plugins output transactions with **negative fake asset IDs** during parsing. The frontend then maps these to real assets via the asset matching UI. This two-phase approach allows import without requiring pre-existing assets.

### Parse result structure
`parse_file()` â†’ returns `(transactions, warnings, extracted_assets)`
- `extracted_assets`: hints about assets found in the file (ISIN, ticker, name) for the matching UI
- `warnings`: non-fatal issues (missing prices, unknown asset types)

### Plugin Version + Stale Detection (Phase 07 Part 2, Revision 2)
Each plugin declares `plugin_version: str` (default `"1.0.0"`). When a plugin is updated:
- `BRIMFileInfo.parse_is_stale = True` if `parsed_plugin_version != registry.current_version`
- Computed lazily in `get_file_info()` â€” no DB write needed
- UI can show a "stale parse" indicator and prompt re-parse

### BRIM is a Parser Only (Phase 07 Part 2, Revision 2)
An earlier v1 implementation added `BRIMCapabilities`, `asset_events` in parse output,
and a `POST /import/commit` endpoint. All were **removed** in Revision 2:
- Capabilities had no consumers
- Asset events (dividends) are the asset provider's job, not the broker parser's
- Commit uses the standard `/transactions/bulk` endpoint â€” no BRIM-specific path

See [[decisions/brim-parser-only]].

### Internal DRY refactoring (Phase 07 Part 2)
- `_build_file_info_from_metadata(meta_path)` â€” unified two identical inner helpers
- `_find_metadata_path(file_id)` â€” DRY metadata path scanning
- `AssetSourceManager.bulk_upsert_events_manual()` renamed to `bulk_upsert_events()`
  (suffix was redundant after `bulk_upsert_events_strict` was removed)

### Test standardization (Phase 07 Part 2)
Merged `TestPluginInterface` (Pattern A) and `TestBRIMPluginsContract` (Pattern B) into
a single `TestBRIMPlugin` class using Pattern B â€” fail-fast at collection time, consistent
with asset/FX provider tests.
