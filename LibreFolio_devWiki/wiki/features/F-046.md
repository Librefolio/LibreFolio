# F-046 — Transaction Model & Bulk API

## What it does
The backend transaction data model and atomic multi-broker bulk API for recording investment
transactions. Types: BUY, SELL, DIVIDEND, INTEREST, FEE, TAX, TRANSFER_IN, TRANSFER_OUT, SPLIT, OTHER.
Each transaction belongs to a broker and references an asset. The API supports multi-broker atomic
batches, dry-run validation, event suggestion, and transfer pair promotion.

## Layer breakdown
**Backend**:
- `backend/app/db/models.py` — `Transaction` model: `asset_id`, `broker_id`, `type`, `date`, `quantity`, `price`, `fees`, `currency`, `related_transaction_id` (self-FK for pairs), `asset_event_id` (FK to AssetEvent)
- `backend/app/api/v1/transactions.py` — all endpoints
- `backend/app/services/transaction_service.py` — `create_bulk`, `update_bulk`, `delete_bulk`, `validate_bulk`, `promote_transfer`

## API Endpoints (Phase 07 Part 3 final)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/transactions/bulk` | Create batch — atomic multi-broker |
| PATCH | `/transactions/bulk` | Update batch — atomic multi-broker |
| DELETE | `/transactions/bulk?ids=...` | Delete batch — atomic multi-broker |
| POST | `/transactions/validate` | Dry-run: mixed create+update+delete, no side effects |
| GET | `/transactions?ids=1,2,...` | Query by IDs (ordered), filtered to accessible brokers |
| GET | `/transactions` | List with filters (broker, asset, type, currency, amount_abs, unlinked, exclude_ids) |
| POST | `/transactions/events/suggest` | Suggest AssetEvents matching TX batch (±0–7 days) |
| POST | `/transactions/transfers/promote` | Promote DEPOSIT/WITHDRAWAL pair → TRANSFER or FX_CONVERSION |

**REMOVED** (Phase 07): `GET /transactions/{tx_id}` — replaced by `GET /transactions?ids=`.

## Atomicity (multi-broker, Phase 07 Part 3)

Endpoints accept items across multiple brokers in one batch:
- Each distinct `broker_id` → EDITOR access check
- `_validate_broker_balances` called per broker
- Any violation → **total rollback** + `rolled_back=True`
- Per-item `status: "success"|"simulated"|"failed"|"not_attempted"`

See [[decisions/multi-broker-atomic-tx]].

## link_uuid / Transfer Semantics (Phase 07 Part 3 Blocco H)

`link_uuid` pairs two transactions with a bidirectional `related_transaction_id` FK.

| Pair type | Same broker? | Enforcement |
|-----------|:------------:|-------------|
| TRANSFER | ❌ different required | validate in `create_bulk` |
| FX_CONVERSION | ✅ intra-broker OK | no restriction |
| DEPOSIT/WITHDRAWAL | optional | produces related_transaction_id for UX |

`POST /transactions/transfers/promote` converts DEPOSIT/WITHDRAWAL → TRANSFER or FX_CONVERSION
atomically (delete pair + create pair with new type + link_uuid).

See [[decisions/tx-link-uuid-semantics]].

## Validate Dry-Run

`POST /transactions/validate` accepts a mixed batch (creates + updates + deletes in same request).
Returns per-item `status: "simulated"|"failed"|"not_attempted"`. Zero DB side effects.
Used by the Staging Modal for live validation (debounced) before final commit.

## events/suggest (Phase 07 Part 3)

`POST /transactions/events/suggest` — for each input transaction, searches for `AssetEvent`
candidates within ±`tolerance_days` (0–7). Used by the Staging Modal to auto-link DIVIDEND
transactions to their corresponding events in the asset's event timeline.

See [[F-051]] for the TX↔AssetEvent link feature.
