# F-046 — Transaction Model & Bulk API

## What it does
The backend transaction data model and atomic multi-broker API for recording investment
transactions. Types: BUY, SELL, DIVIDEND, INTEREST, DEPOSIT, WITHDRAWAL, FEE, TAX, TRANSFER, FX_CONVERSION, ADJUSTMENT.
Each transaction belongs to a broker and optionally references an asset. The API supports multi-broker atomic
batches, dry-run validation, event suggestion, transfer pair promotion, and server-driven type metadata.

## Layer breakdown
**Backend**:
- `backend/app/db/models.py` — `Transaction` model
- `backend/app/api/v1/transactions.py` — all endpoints
- `backend/app/services/transaction_service.py` — `execute_batch` (unified pipeline), `promote_transfer`
- `backend/app/schemas/transactions.py` — `TXMixedBatch`, `TXBatchResponse`, `TXTypeMetadata`, `TXTypesResponse`

## API Endpoints (Phase 07 Part 4 Round 4 — Unified Batch Pipeline)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/transactions/validate` | Dry-run: unified lenient parse + balance walk, always rollback |
| POST | `/transactions/commit` | Unified commit: lenient parse + balance walk, commit if clean |
| GET | `/transactions` | List with filters (broker, asset, type, currency, etc.) |
| GET | `/transactions?ids=1,2,...` | Query by IDs (ordered) |
| GET | `/transactions/types` | Returns `TXTypesResponse` with type metadata + event types |
| POST | `/transactions/events/suggest` | Suggest AssetEvents matching TX batch (±0–7 days) |
| POST | `/transactions/transfers/promote` | Promote DEPOSIT/WITHDRAWAL pair → TRANSFER or FX_CONVERSION |

**REMOVED** (Round 4): `POST /bulk`, `PATCH /bulk`, `DELETE /bulk` — replaced by unified `/validate` + `/commit`.

Both `/validate` and `/commit` accept `TXMixedBatch { creates: List[dict], updates: List[dict], deletes: List[int] }` and return `TXBatchResponse { committed: bool, issues: List[TXValidationIssue], results?: List[TXBatchResultItem] }`. See [[decisions/unified-batch-pipeline]].

## Unified Pipeline (`execute_batch`)

Single entry point for all tx mutations:
1. **Lenient per-row parse** via `_parse_lenient()` — `Model.model_validate(raw)` per row in try/except
2. **Access check** (EDITOR role per broker)
3. **Apply** deletes → updates → creates
4. **Balance walk** per affected broker
5. Schema + business + balance errors coexist in `issues[]`

## Server-Driven Type Rules (`GET /transactions/types`)

Returns `TXTypesResponse` with:
- `TXTypeMetadata`: `icon_slug`, `doc_slug`, `FieldMode` (asset/cash/quantity), `SignRule` (quantity/cash), `pair_form_layout`, `requires_link`, `event_compatible`
- `EventTypeMetadata`: `code`, `name`, `emoji`, `compatible_tx_types`

See [[decisions/server-driven-type-rules]].
