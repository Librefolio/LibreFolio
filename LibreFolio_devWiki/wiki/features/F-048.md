# F-048 — Transaction Modals (Form / Bulk / Delete / Promote Wizard)

## What it does
Four specialised modal components for creating, editing, duplicating, deleting, and promoting transactions. Replace the original monolithic `TransactionStagingModal`. All validate via `POST /transactions/validate` and commit via `POST /transactions/commit` using the unified batch pipeline.

## Layer breakdown
**Frontend**: Four components in `frontend/src/lib/components/transactions/`:
- `TransactionFormModal.svelte` — single-row CRUD (create/edit/duplicate/view) + dual-form mode for paired transactions
- `TransactionBulkModal.svelte` — **Mode-less** DataTable-based unified batch editor: create + edit + delete in one modal. See [[decisions/bulkmodal-mode-removal]]
- `TransactionDeleteModal.svelte` — Rich single/paired delete confirmation with 3 layouts (A/B/C), server-side validation, error resolution
- `PromotePairWizardModal.svelte` — 3-step wizard for promoting DEPOSIT/WITHDRAWAL pairs

## Architecture (Phase 7 Part 4 Rounds 3–6)

### TransactionFormModal
- **Modes**: create, edit, duplicate, view
- **Dual-form mode** (R6-B): `pair_form_layout` from `transactionTypeStore` drives 3 layouts: FX (split currencies), Transfer Asset (split brokers, shared asset), Transfer Cash (split brokers, shared amount). See [[decisions/dual-transaction-form-design]]
- **Sectioned form**: Required (Date, Type, Qty, Cash, Asset, Broker) → Optional (tags, description) → Advanced (event link, cost_basis_override, pair partner)
- **Type-driven field adaptation**: all fields adapt visibility/labels/sign hints from server-driven type rules ([[decisions/server-driven-type-rules]])
- **Auto-sign negation**: user enters positive numbers when type requires negative (SELL qty, BUY cash) — auto-negated in `collectCreate()`/`collectUpdate()`
- **Unsaved changes guard**: JSON snapshot comparison triggers ConfirmModal on close/escape
- **Tags autocomplete**: client-side aggregation from loaded transactions + drafts
- **PATCHABLE_FIELDS allowlist** (Bugfix 3): update payloads strip immutable fields via shared `txPayloadHelpers.ts` allowlist
- **Type swap** (Bugfix 3): backend supports type change within swap groups (BUY↔SELL, DEPOSIT↔WITHDRAWAL, DIVIDEND↔INTEREST, TAX↔FEE). Paired types locked. Sign-flip on quantity/cash automatic.
- **Shared payload helpers** (Plan A): `txPayloadHelpers.ts` — single source for PATCHABLE_FIELDS, applySignRules, buildUpdateDiff, fieldEq. Eliminates FormModal/BulkModal logic duplication (see [[problems/dual-form-collect-duplication]])
- **Broker access** (Plan B): form dropdown shows only `getEditableBrokers()` (OWNER/EDITOR). View mode for VIEWER/inaccessible partners shows locked placeholder. `canEdit` prop gates ✏ button.

### TransactionBulkModal (mode-less unified batch editor)
- **Mode-less design** (Plan B23 follow-up): no `mode` prop. Each row's role inferred from its data — `tx.id > 0` = DB row (edit), `tx.id ≤ 0` = new draft (create). See [[decisions/bulkmodal-mode-removal]]
- Built on `DataTable` with editable cells
- **Row states**: `new` (green CREATE), `original` (unchanged, skipped), `edited` (amber UPDATE), `delete` (red DELETE with line-through)
- **Bulk delete flow**: opened with `initialStatus: 'delete'` — all rows start marked for deletion, user can restore/edit/clone/add from the same batch
- **Confirm edit on delete**: editing a row marked for deletion triggers a ConfirmModal ("Restore & Edit?") before opening FormModal
- **Paired row split hint**: InfoBanner shown when paired rows are marked for deletion, suggesting Split to unlink before partial delete
- **PickerModal**: "Search & add" button visible when `allMainRows.length > 0`. Guard for non-editable rows (VIEWER broker) with disabled checkbox (⊘), tooltip, select-all skip. Double-click (desktop) and long-press (mobile) toggle selection
- **Auto-validate**: disabled when N > 50 rows; manual `⚡ Validate now` always available. See [[concepts/validate-scheduler-pattern]]
- **i18n error display**: structured error codes resolved via `resolveValidationMessage.ts`. See [[concepts/resolve-validation-message-pattern]]
- **Anti-bounce**: 10s deduplication on validate+commit when draftKey unchanged
- **Commit failure banner**: red ⛔ banner with categorised issues (field errors vs balance errors with client-side row attribution)
- **CASH_TRANSFER** (Bugfix 1): `collectDualCreates('transfer_cash')` emits CASH_TRANSFER+CASH_TRANSFER (not WITHDRAWAL+DEPOSIT). See [[decisions/cash-transfer-split-promote]]
- **Paired edit fix** (R7-C1, Plan A): `patchDualRowFromForm()` preserves partner id/status/original for DB rows; `collectDualCreates()` reuses existing `link_uuid` in edit mode

### TransactionDeleteModal (3-layout rich confirmation)
- **Layout A** (standalone): full detail table — Date, Type (with icon), Qty, Cash, Asset (with icon), Broker (BrokerBadge + role), Tags (colored), Description (with tooltip)
- **Layout B** (paired full): side-by-side From/To table with giver/receiver columns, asset icons, broker badges. Split hint below
- **Layout C** (paired blocked/inaccessible): partner broker locked — shows Lock icon, broker name, "not accessible" message. Delete button hidden
- **Validation**: `errors[]` prop for backend error display via `resolveIssueMessage()`, `validating`/`validated` props, `onValidate` callback for "⚡ Validate now" button
- **Success toast**: HTML with transaction type icon, broker icon, role SVG (`getRoleSvgHtml`), bold broker name. Paired variant shows from→to
- **Broker access** (Plan B): Layout C shown when user lacks EDITOR on partner broker. Split hint in Layout B. See [[decisions/broker-access-min-paired]]

### TagInput.svelte (Bugfix 3 + Round 6)
- New chip input component with autocomplete dropdown from available tags
- Keyboard navigation: ArrowDown/Up + Enter on highlight
- Colored chips via `getStringColor()` / `getStringBadgeStyle()`
- Anti-bounce via `relatedTarget` check (replaces fragile setTimeout)
- Chip navigation: ←→ + Delete/Backspace on selected chip with `ring-2` highlight
