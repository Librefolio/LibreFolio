# F-048 — Transaction Modals (Form / Bulk / Promote Wizard)

## What it does
Three specialised modal components for creating, editing, duplicating, and promoting transactions. Replace the original monolithic `TransactionStagingModal`. All validate via `POST /transactions/validate` and commit via `POST /transactions/commit` using the unified batch pipeline.

## Layer breakdown
**Frontend**: Three components in `frontend/src/lib/components/transactions/`:
- `TransactionFormModal.svelte` — single-row CRUD (create/edit/duplicate/view) + dual-form mode for paired transactions
- `TransactionBulkModal.svelte` — DataTable-based multi-row bulk editor with editable cells, ColumnVisibilityToggle, mixed row states (new/edit/delete)
- `PromotePairWizardModal.svelte` — 3-step wizard for promoting DEPOSIT/WITHDRAWAL pairs

## Architecture (Phase 7 Part 4 Rounds 3–5)

### TransactionFormModal
- **Modes**: create, edit, duplicate, view
- **Dual-form mode** (R6-B): `pair_form_layout` from `transactionTypeStore` drives 3 layouts: FX (split currencies), Transfer Asset (split brokers, shared asset), Transfer Cash (split brokers, shared amount). See [[decisions/dual-transaction-form-design]]
- **Sectioned form**: Required (Date, Type, Qty, Cash, Asset, Broker) → Optional (tags, description) → Advanced (event link, cost_basis_override, pair partner)
- **Type-driven field adaptation**: all fields adapt visibility/labels/sign hints from server-driven type rules ([[decisions/server-driven-type-rules]])
- **Auto-sign negation**: user enters positive numbers when type requires negative (SELL qty, BUY cash) — auto-negated in `collectCreate()`/`collectUpdate()`
- **Unsaved changes guard**: JSON snapshot comparison triggers ConfirmModal on close/escape
- **Tags autocomplete**: client-side aggregation from loaded transactions + drafts

### TransactionBulkModal
- Built on `DataTable` with editable cells (`EditableTextCell`, `EditableSelectCell`, `CustomCell` wrapping `BrokerSearchSelect`, `SingleDatePicker`, `TransactionTypeSearchSelect`, `AssetSelect`, `CompactCashCell`)
- **Mixed row states**: `new` (green), `edit` (blue), `delete` (red) — committed together via unified pipeline
- **Auto-validate**: disabled when N > 50 rows; manual `⚡ Validate now` always available. See [[concepts/validate-scheduler-pattern]]
- **i18n error display**: structured error codes resolved via `resolveValidationMessage.ts` with store-backed ID→name resolution. See [[concepts/resolve-validation-message-pattern]]
- **Anti-bounce**: 10s deduplication on validate+commit when draftKey unchanged
- **Commit failure banner**: red ⛔ banner with categorised issues (field errors vs balance errors with client-side row attribution)

### PromotePairWizardModal
- 3-step wizard: pair picker → choose type (TRANSFER/FX_CONVERSION with greyed-out incompatible options) → details (asset+quantity for TRANSFER)
- 3 sources per slot: in-bulk, saved transactions, create-new (opens nested FormModal)
- Uses `POST /transactions/transfers/promote` endpoint

### Shared utilities
- `CompactCashCell.svelte` — generic numeric input + CurrencySearchSelect compact
- `TransactionTypeSearchSelect.svelte` — SearchSelect with TX type PNG icon + translated label
- `useValidateScheduler.svelte.ts` — validate scheduling factory
- `resolveValidationMessage.ts` — i18n error resolution
- `formatDecimalForDisplay.ts` — trims trailing zeros from backend decimals
